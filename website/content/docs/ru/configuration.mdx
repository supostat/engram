# Конфигурация

Engram разделяет состояние на две локации:

- **Глобальная** (`~/.engram/`) — `engram.toml` (базовый слой конфигурации, хранит API-ключи и дефолты), `models/` (ONNX-артефакты, переиспользуемые между проектами).
- **Per-project** (`<project>/.engram/`) — `engram.toml` (project-local переопределения конфигурации), `engram.db` (SQLite) и `engram.sock` (Unix-сокет). Discovery поднимается вверх от cwd в поисках директории `.engram/`, как `.git`.

## Послойная конфигурация

`engram.toml` разрешается **послойно**, с приоритетом `env > project-local > global`:

1. **Глобальный** `~/.engram/engram.toml` — базовый слой. Хранит API-ключи и дефолты. `embedding.api_key` и `llm.api_key` всегда берутся отсюда.
2. **Project-local** `<project>/.engram/engram.toml` — находится подъёмом вверх от текущей директории в поисках маркера `.engram/`, как `.git`. Если он есть, то глубоко мёржится поверх глобального слоя поле за полем: заданные поля переопределяют глобальное значение, незаданные — наследуются от глобального. Вложенные таблицы (`[embedding]`, `[hnsw]`, …) мёржатся рекурсивно. **`api_key`, заданный в project-local файле, игнорируется** — секреты всегда берутся из глобального конфига.
3. **Переменные окружения `ENGRAM_*`** — применяются последними, с наивысшим приоритетом. Они перезаписывают значения из любого файлового слоя.

CLI-флаг `--config <path>`, если передан, заменяет глобальный слой указанным файлом. Если конфиг-файла нет вовсе — применяются встроенные значения по умолчанию.

`engram init` пишет минимальный, полностью закомментированный project-local шаблон `engram.toml` в `<project>/.engram/`. Шаблон содержит только несекретные поля-переопределения, все закомментированы — нетронутый файл наследует глобальный конфиг дословно.

## Разрешение путей

Пути к базе и сокету разрешаются в рантайме, а не из конфига:

| Ресурс | Порядок приоритета |
|--------|---------------------|
| База данных | `ENGRAM_DB_PATH` env → `<project>/.engram/engram.db` (per-project, найден подъёмом вверх от cwd) → `~/.engram/memories.db` (legacy fallback) |
| Сокет | `ENGRAM_SOCKET_PATH` env → `<project>/.engram/engram.sock` (если существует project `.engram/`) → `server.socket_path` из `engram.toml` (с раскрытием `~`) |

Значения `database.path` и `server.socket_path` в `engram.toml` остаются как fallback'и для случаев, когда маркер `.engram/` не найден при подъёме. `engram server` прерывает запуск, если найден legacy `~/.engram/engram.db`, но нет per-project `.engram/engram.db` — в сообщении об ошибке указана команда `engram migrate`.

## Структура engram.toml

```toml
[database]
# Только fallback — рантайм предпочитает per-project .engram/engram.db.
path = "~/.engram/memories.db"

[embedding]
provider = "voyage"         # voyage | ollama | deterministic
model = "voyage-4"
# host = "http://localhost:11434"   # ollama only; ENGRAM_OLLAMA_HOST overrides
# api_key = "..."           # или через ENGRAM_VOYAGE_API_KEY
# output_dimension = 1024   # опционально; Voyage-4 Matryoshka (256/512/1024/2048). Должно совпадать с [hnsw].dimension.
hyde_threshold = 0          # 0 = HyDE выключен (по умолчанию); N>0 включает HyDE для запросов короче N слов

[llm]
provider = "openai"         # openai | ollama | local
model = "gpt-4o-mini"
# host = "http://localhost:11434"   # ollama only; ENGRAM_OLLAMA_HOST overrides
# api_key = "..."           # или через ENGRAM_OPENAI_API_KEY

[server]
# Только fallback — рантайм предпочитает per-project .engram/engram.sock.
socket_path = "~/.engram/engram.sock"
reindex_interval_secs = 3600

[hnsw]
max_connections = 16
ef_construction = 200
ef_search = 40
dimension = 1024

[consolidation]
stale_days = 90
min_score = 0.3
fts_similarity_floor = 0.0

[trainer]
trainer_binary = "engram-trainer"
trainer_timeout_secs = 300
models_path = "~/.engram/models"
```

## Секции

### database

| Параметр | По умолчанию | Описание |
|----------|-------------|----------|
| `path` | `~/.engram/memories.db` | Fallback-путь к SQLite (в рантайме перекрывается per-project `.engram/engram.db` — см. [Разрешение путей](#разрешение-путей)) |

### embedding

| Параметр | По умолчанию | Описание |
|----------|-------------|----------|
| `provider` | `voyage` | Провайдер эмбеддингов: `voyage`, `ollama` или `deterministic` |
| `model` | `voyage-4` | Модель эмбеддингов |
| `api_key` | — | API-ключ (предпочтительно через переменную окружения) |
| `dimension` | `1024` | Размерность вектора эмбеддинга |
| `output_dimension` | — | Опциональная Voyage-4 Matryoshka-усечение (256/512/1024/2048). Если задана, должна совпадать с `[hnsw].dimension`. Опусти — API использует значение по умолчанию (1024). |
| `hyde_threshold` | `0` | HyDE opt-in: `0` отключает HyDE; `N > 0` включает его для запросов короче `N` слов |

> Смена `model` на уже заполненной базе приводит к ошибке `[6020] EmbeddingModelMismatch` при следующем старте `engram server`. Запусти один раз [`engram reembed`](/docs/cli#engram-reembed), чтобы пересчитать все векторы с активным провайдером, и перезапусти демон.

Провайдер `deterministic` не требует API-ключа и генерирует детерминированные эмбеддинги из текста. Подходит для тестирования.

Провайдер `ollama` обращается к локальному демону [Ollama](https://ollama.com) (host по умолчанию `http://localhost:11434`, переопределяется через `ENGRAM_OLLAMA_HOST`) и не требует API-ключа — например, `model = "qwen3-embedding:0.6b"` при 1024-dim. Переключение существующей базы на него — однократный [`engram reembed`](/docs/cli#engram-reembed).

При `hyde_threshold > 0` короткие запросы запускают LLM-вызов, который генерирует гипотетическое воспоминание, и эмбеддинг строится по гипотезе вместо сырого запроса. Trade-off: cache miss добавляет ~1.5s на запрос, но улучшает recall на коротких запросах; кеш ключуется по **оригинальному запросу**, поэтому повторные одинаковые запросы попадают в кеш без дополнительного LLM-вызова.

### llm

| Параметр | По умолчанию | Описание |
|----------|-------------|----------|
| `provider` | `openai` | Провайдер LLM: `openai`, `ollama` или `local` |
| `model` | `gpt-4o-mini` | Модель для генерации текста |
| `api_key` | — | API-ключ (предпочтительно через переменную окружения) |

Провайдер `local` использует ONNX-модель из `trainer.models_path` для локальной генерации без API-вызовов.

Провайдер `ollama` запускает LLM-функции (judge, HyDE, консолидация) на локальной chat-модели Ollama (например, `model = "qwen3:4b"`) без API-ключа; если демон недоступен, judge откатывается на эвристики.

### server

| Параметр | По умолчанию | Описание |
|----------|-------------|----------|
| `socket_path` | `~/.engram/engram.sock` | Fallback-путь к Unix-сокету (в рантайме перекрывается per-project `.engram/engram.sock` — см. [Разрешение путей](#разрешение-путей)) |
| `reindex_interval_secs` | `3600` | Интервал переиндексации (секунды, 0 = выключено) |

### hnsw

| Параметр | По умолчанию | Описание |
|----------|-------------|----------|
| `max_connections` | `16` | Максимум соединений в графе HNSW |
| `ef_construction` | `200` | Точность при построении индекса |
| `ef_search` | `40` | Точность при поиске |
| `dimension` | `1024` | Размерность векторов (должна совпадать с embedding.dimension) |

### consolidation

| Параметр | По умолчанию | Описание |
|----------|-------------|----------|
| `stale_days` | `90` | Порог устаревания записей (дни) |
| `min_score` | `0.3` | Минимальный score для кандидатов консолидации |
| `fts_similarity_floor` | `0.0` | Минимальный средний &#124;bm25&#124; rank FTS-группы дубликатов в consolidation preview (та же шкала, что `similarity` у `fts`-групп, не cosine). `0.0` выключает фильтр; exact-группы никогда не фильтруются. Значение должно быть конечным и ≥ 0. |

### trainer

| Параметр | По умолчанию | Описание |
|----------|-------------|----------|
| `trainer_binary` | `engram-trainer` | Путь или имя бинарника trainer |
| `trainer_timeout_secs` | `300` | Таймаут выполнения trainer (секунды) |
| `models_path` | `~/.engram/models` | Директория для ONNX-моделей |

## Переменные окружения

| Переменная | Перезаписывает |
|-----------|---------------|
| `ENGRAM_VOYAGE_API_KEY` | `embedding.api_key` (при provider=voyage) |
| `ENGRAM_OPENAI_API_KEY` | `llm.api_key` (при provider=openai) |
| `ENGRAM_OLLAMA_HOST` | `embedding.host` / `llm.host` (при provider=ollama) |
| `ENGRAM_DB_PATH` | `database.path` |
| `ENGRAM_SOCKET_PATH` | `server.socket_path` |
| `ENGRAM_EMBEDDING_MODEL` | `embedding.model` |
| `ENGRAM_LLM_MODEL` | `llm.model` |
| `ENGRAM_TRAINER_BINARY` | `trainer.trainer_binary` |
| `ENGRAM_TRAINER_TIMEOUT` | `trainer.trainer_timeout_secs` |
| `ENGRAM_MODELS_PATH` | `trainer.models_path` |

## Пути по умолчанию

```
~/.engram/                       # глобальная — общая для всех проектов
  engram.toml                    # конфигурация (API-ключи, дефолты)
  models/                        # ONNX-модели trainer

<project>/.engram/               # per-project — создаётся через `engram init`
  engram.toml                    # project-local переопределения (глубокий мёрж поверх глобального)
  engram.db                      # SQLite база данных
  engram.sock                    # Unix-сокет сервера
  *.hnsw                         # сериализованные HNSW-индексы
  AGENT.md                       # упакованные инструкции для агента
```

## Инициализация

Команда `engram init` создаёт директорию `.engram/` в текущем рабочем каталоге (с `engram.db`, минимальным закомментированным project-local `engram.toml`, упакованным `AGENT.md`, `.mcp.json` и записью `.engram/` в `.gitignore`). При первом запуске также пишет дефолтный `~/.engram/engram.toml`, если его нет. Существующие файлы никогда не перезаписываются.

Чтобы импортировать данные из предыдущего глобального инсталла (`~/.engram/engram.db`), см. [`engram migrate`](/ru/docs/cli#engram-migrate).
